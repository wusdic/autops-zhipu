# AUTOPS 全局约束与设计优先级

> 项目名称：autops  
> 文档状态：current  
> 优先级：最高  
> 适用范围：本项目所有设计、编码、测试、文档、部署、验收、重构和后续演进。  
> 核心原则：先把设计做到足够完整、清晰、可落地，再开始编码。任何实现不得绕过本文件约束。

---

## 1. 项目起点

AUTOPS 按照全新项目规划和建设，默认起点为空仓库。所有功能、目录、模块、数据模型、接口、前端页面、部署脚本、测试和文档都从目标架构出发重新设计。

本项目不以任何已有实现为边界，不迁就任何既有目录、接口、数据库结构或页面结构。所有建设动作只服从本项目目标架构、领域模型、全局约束和开发计划。

---

## 2. 最高目标

AUTOPS 的最高目标是建设一个面向私有化部署场景的 **自治运维操作系统**。

它不是普通监控平台，不是工单系统，不是脚本平台，也不是简单 AI 助手。它要把资产、配置、状态、事件、告警、日志、策略、自动化执行、AI 分析、工单协同、知识沉淀和平台自身治理整合成一个可持续演进的闭环系统。

核心闭环：

```text
资源发现
  ↓
资产纳管
  ↓
配置和凭证治理
  ↓
状态、指标、日志、配置事实采集
  ↓
事件识别
  ↓
告警收敛
  ↓
影响分析
  ↓
AI 诊断
  ↓
策略决策
  ↓
自动化执行
  ↓
实时日志
  ↓
结果验证
  ↓
回滚 / 升级 / 关闭
  ↓
工单复盘
  ↓
知识沉淀
  ↓
策略优化
```

## 3. 设计优先于编码

### 3.1 不允许直接编码

在完成以下设计文件前，不允许进入正式编码：

1. 总体目标架构。
2. 数据架构。
3. 后端领域架构。
4. 前端工作流架构。
5. 安全架构。
6. API 契约。
7. 数据库模型。
8. 事件模型。
9. 策略模型。
10. 自动化执行模型。
11. AIops 工具调用边界。
12. 测试策略。
13. 部署与升级方案。
14. 文档治理方案。

### 3.2 编码前必须先完成设计审查

每个领域模块编码前，必须先完成：

- 模块职责说明。
- 输入输出边界。
- 数据模型。
- API 草案。
- 事件定义。
- 权限控制。
- 审计要求。
- 异常处理。
- 测试用例。
- 与其他模块的依赖关系。

### 3.3 任何代码必须可追溯到设计

每个重要模块、接口、表、事件、策略、执行动作，都必须能在文档中找到设计依据。如果代码实现中出现文档没有定义的新概念，应先更新文档，再实现。

## 6. MVP 的定位

MVP 不是项目目标，也不是开发中心。MVP 只是用于验证整体架构是否真实可运行的验收场景。

项目开发必须按完整平台规划推进，而不是围绕某个单一场景写专项代码。MVP 场景只能调用平台通用能力完成，不能为 MVP 写旁路逻辑、特殊接口、硬编码规则或一次性脚本。

第一阶段推荐使用多个标准化场景验证平台闭环，包括：

1. Linux 磁盘空间异常。
2. Windows 服务异常。
3. Web 服务端口不可达。
4. 数据库连接异常或连接数过高。
5. 证书即将过期。
6. 采集器离线。
7. 自动化执行失败。

这些场景必须通过统一资产、配置、采集、事件、告警、策略、执行、日志、AI 和知识库能力完成。

## 7. 国产化与私有化兼容原则

### 7.1 组件选择原则

组件选择必须满足：

1. 简单优先。
2. 私有化部署友好。
3. 离线部署可行。
4. 后续国产化迁移阻力低。
5. 不依赖复杂云服务。
6. 不绑定单一厂商。
7. 不依赖特定数据库方言。
8. 可在 x86 和 ARM 环境部署。
9. 可容器化，也能支持物理机部署。

### 7.2 数据库原则

关系型数据库默认优先选择 MySQL/MariaDB 兼容路线，同时通过 SQLAlchemy 等抽象层保留 PostgreSQL、openGauss、人大金仓、达梦等迁移空间。

设计要求：

- 不使用不可迁移的数据库特性作为核心能力。
- 避免强依赖 PostgreSQL JSONB、数组、特定索引语法。
- 复杂结构可使用标准表结构，不滥用 JSON。
- 必须使用 Alembic 或等价 migration 机制。
- 所有 SQL 必须尽量符合通用 SQL。
- 数据访问必须通过 repository 层隔离。
- 业务代码不得直接散落原生 SQL。

### 7.3 中间件原则

推荐基础组件：

- Redis 兼容组件：缓存、分布式锁、轻量队列。
- VictoriaMetrics 或 Prometheus：指标存储。
- Loki 或文件/对象存储 + 索引：日志存储，第一阶段可先用关系库保存执行日志索引和日志分片。
- MinIO 或 S3 兼容对象存储：附件、报告、归档。
- Qdrant、Chroma 或可替换向量存储：知识向量检索。
- vLLM / Ollama 等本地模型服务：大模型推理。

所有组件必须通过 adapter 或 service 层访问，避免业务代码直接绑定具体实现。

## 8. 大模型配置原则

本地已有 vLLM，大模型暂时使用 Qwen3.5 0.8B，但代码不得写死模型名称、服务地址或参数。

模型必须通过配置文件管理，例如：

```yaml
llm:
  provider: vllm
  base_url: http://127.0.0.1:8000/v1
  model: qwen3.5-0.8b
  timeout_seconds: 60
  max_tokens: 2048
  temperature: 0.2

embedding:
  provider: local
  model: configurable-embedding-model
  base_url: http://127.0.0.1:8001
```

要求：

- 模型服务地址可配置。
- 模型名称可配置。
- 推理参数可配置。
- Prompt 模板可配置。
- 工具调用权限可配置。
- 不同场景可配置不同模型。
- 模型不可用时平台应降级，而不是崩溃。
- AI 不得绕过策略中心和审批机制直接执行高风险动作。

## 9. 安全边界

### 9.1 凭证安全

- 所有密码、Token、SSH Key、API Key 必须加密存储。
- 推荐使用 AES-256-GCM 或可国产化替换的对称加密算法。
- 主密钥不得写入代码和数据库。
- 凭证读取、测试、绑定、使用必须审计。
- AI 不得读取明文凭证。
- 导出数据不得包含明文凭证。

### 9.2 自动化安全

- 所有执行必须有 execution_id。
- 所有执行必须有审计日志。
- 所有执行必须有实时日志。
- 高危动作必须经过策略判断。
- 生产环境高危动作必须人工确认。
- 支持 dry-run。
- 支持并发锁。
- 支持最大影响面控制。
- 支持回滚点。
- 支持执行后验证。
- 支持高危命令黑名单。
- 支持可自动执行动作白名单。

### 9.3 AI 安全

AI 可以分析、解释、推荐、生成计划和知识草稿。AI 不得绕过权限、审批、策略、安全边界和执行引擎。

AI 工具调用必须记录：

- tool_call_id。
- 输入。
- 输出。
- 触发来源。
- 用户确认。
- 策略校验结果。
- 执行结果。
- 审计日志。

## 10. 文档治理原则

AUTOPS 采用 Docs as Code。

要求：

1. 文档与代码在同一仓库。
2. 架构变更必须有 ADR。
3. API 文档应由 OpenAPI 自动生成。
4. 修改领域代码必须同步修改领域文档。
5. 修改 API 必须同步更新 API 契约。
6. 修改部署必须同步更新部署文档。
7. 修改安全边界必须同步更新安全文档。
8. 文档过期必须归档。
9. README 只能作为入口，不承载详细设计。
10. 任何临时结论必须进入正式文档或归档，不得长期散落。

---

## 11. 验收原则

平台是否完成，不看代码量，不看页面数量，而看是否满足：

1. 架构是否一致。
2. 领域边界是否清晰。
3. 数据模型是否可演进。
4. API 是否规范。
5. 前端流程是否符合运维真实工作。
6. 是否能自动发现问题。
7. 是否能自动分析问题。
8. 是否能安全执行处置动作。
9. 是否有实时日志。
10. 是否能验证结果。
11. 是否能失败回滚或升级工单。
12. 是否能沉淀知识。
13. 是否有审计和安全边界。
14. 是否能私有化部署、离线部署和升级回滚。
15. 是否能通过自动化测试验证核心链路。
