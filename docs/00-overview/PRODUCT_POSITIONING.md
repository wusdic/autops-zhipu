## 1. 产品愿景与使命

### 1.1 愿景

**让运维从被动响应走向自治闭环。**

AUTOPS 致力于将运维工作从碎片化的工具堆砌、人工判断和被动救火，转变为一个从发现到知识沉淀的完整自治闭环——系统能够自动感知环境变化、智能分析根因、安全执行处置、验证结果并持续优化。

### 1.2 使命

构建一个 **面向私有化部署场景的自治运维操作系统**，将资产纳管、配置治理、状态感知、事件识别、告警收敛、AI 根因分析、策略安全执行、实时日志、失败回滚和知识沉淀整合为统一平台。

AUTOPS 不是监控平台、工单系统、脚本工具或 AI 问答助手中的任何单一工具，而是将以上能力融合为一个可持续演进的闭环系统。

### 1.3 核心闭环

```text
资源发现 → 资产纳管 → 配置治理 → 状态采集 → 事件识别 → 告警收敛
→ 影响分析 → AI 诊断 → 策略决策 → 自动化执行 → 实时日志
→ 结果验证 → 回滚/升级/关闭 → 知识沉淀 → 策略优化
```

这个闭环不是理想化的理论模型，而是 AUTOPS 的最小运转单元。每一条链路都必须真实可运行、可验证、可审计。

## 2. 目标用户

### 2.1 运维团队（核心用户）

- **职责**：日常基础设施运维、故障响应、变更执行、巡检保障。
- **痛点**：工具碎片化、告警风暴、根因定位慢、处置依赖经验、知识流失。
- **AUTOPS 价值**：统一纳管所有资源，自动收敛告警，AI 辅助根因分析，策略驱动安全执行，处置经验自动沉淀为知识。

### 2.2 安全团队

- **职责**：安全基线管理、漏洞跟踪、合规审计、权限控制。
- **痛点**：缺乏统一审计视图、凭证管理分散、执行动作不可追溯。
- **AUTOPS 价值**：全链路审计日志、凭证加密管理与使用审计、高危动作强制审批、动态脱敏、RBAC/ABAC 权限控制。

### 2.3 开发团队

- **职责**：应用服务部署、运行状态感知、故障定位协作。
- **痛点**：缺乏环境全局视图、与应用相关的基础设施故障感知滞后。
- **AUTOPS 价值**：应用与基础设施关联视图、日志与指标统一检索、AI 日志解释、故障影响范围自动分析。

### 2.4 管理层

- **职责**：运维效率评估、风险管控、合规要求、预算与资源决策。
- **痛点**：运维效率难以量化、缺乏统一报表、安全合规审计成本高。
- **AUTOPS 价值**：运维健康仪表盘、策略成功率与告警压缩率统计、知识沉淀与团队经验可视化、全链路审计支撑合规。

## 3. 核心价值主张

### 10 大核心价值

| # | 核心价值 | 说明 |
|---|---------|------|
| 1 | **自动发现纳管** | 自动发现 IT 资源（主机、网络设备、数据库、中间件、应用等）并纳入统一管理，支持 Agentless、Agent 和边缘采集器三种模式。 |
| 2 | **实时感知** | 持续采集设备、服务、应用、数据库、中间件和平台自身状态，实时推送状态变更。 |
| 3 | **统一管理** | 统一管理采集配置、凭证、阈值、策略、脚本、通知和 AI 模板，支持多层级配置继承与版本化。 |
| 4 | **事件识别** | 自动识别状态异常、配置漂移、日志命中、执行失败等事件，标准化处理并路由至正确处理链路。 |
| 5 | **告警收敛** | 告警去重、关联、抑制、静默、升级，显著减少告警风暴，自动判断影响范围。 |
| 6 | **AI 根因分析** | 基于多源证据（资产信息、指标趋势、事件链、日志上下文、配置变更、历史案例）进行结构化根因分析。 |
| 7 | **策略安全执行** | 根据策略驱动自动化动作，支持 dry-run、审批、并发锁、分批执行、灰度执行、高危命令黑名单。 |
| 8 | **实时日志回传** | 执行过程实时日志流推送，支持 WebSocket 实时查看，执行结果完整可追溯。 |
| 9 | **失败回滚** | 执行失败自动回滚或升级工单，支持回滚点、执行后验证、生产环境二次确认。 |
| 10 | **知识沉淀** | 处置经验自动沉淀为结构化知识，支持向量检索、知识草稿审核、策略优化反馈。 |

### 价值承诺

- **从碎片到闭环**：不再需要在 5-10 个工具之间来回切换，一个平台覆盖发现→感知→分析→执行→沉淀全链路。
- **从经验到知识**：运维经验不再只存在于个人大脑中，每一次处置都沉淀为可检索、可复用的结构化知识。
- **从手动到自治**：从人工判断和手动执行，演进为 AI 辅助决策 + 策略驱动安全执行 + 自动验证回滚。
- **从公有云依赖到私有化自主**：完整支持私有化部署、离线部署、国产化兼容，不依赖任何公有云服务。

### 4.2 核心差异总结

**AUTOPS 不是以下任何单一工具：**

- **不是 Zabbix/Prometheus**：AUTOPS 不仅采集指标和生成告警，还将告警纳入完整的分析→决策→执行→验证闭环。
- **不是 Ansible/SaltStack**：AUTOPS 不仅执行脚本和 Playbook，还提供策略安全护栏、审批机制、实时日志、失败回滚和执行审计。
- **不是 PagerDuty/AlertManager**：AUTOPS 不仅路由告警，还自动分析根因、推荐动作、驱动执行并沉淀知识。
- **不是 AI 问答助手**：AUTOPS 不仅提供 AI 建议，还将 AI 输出纳入策略校验、执行审计和安全边界，AI 不可绕过权限和审批。
- **不是 CMDB**：AUTOPS 不仅管理资产台账，还通过自动发现保持资产实时性，并与状态、事件、策略、执行深度关联。
- **不是工单系统**：AUTOPS 不仅管理工单流程，还将工单与告警、事件、执行、日志、AI 分析全链路关联，并支持工单复盘转知识。

**AUTOPS 是以上能力的统一闭环整合。**

## 6. 适用场景

### 6.1 企业私有化运维

- **场景**：中大型企业自建数据中心或混合云环境，要求运维平台完全部署在内部网络。
- **需求**：不依赖公有云、不外传数据、支持国产化替代。
- **AUTOPS 优势**：从架构设计阶段就以私有化部署为核心目标，支持单机、Docker Compose、多节点和物理机部署。

### 6.2 离线/隔离网络环境

- **场景**：政府、金融、能源等行业存在网络隔离要求，运维平台必须在完全离线环境运行。
- **需求**：离线安装、离线升级、无外部依赖。
- **AUTOPS 优势**：所有组件支持离线部署，本地大模型推理（vLLM/Ollama），不依赖任何外部 API 或云服务。

### 6.3 国产化替代

- **场景**：需要将现有运维体系迁移到国产化基础设施（国产数据库、国产操作系统、国产硬件）。
- **需求**：数据库兼容、操作系统兼容、硬件架构兼容（x86 + ARM）。
- **AUTOPS 优势**：通过 SQLAlchemy 抽象层保留多数据库迁移空间，避免不可迁移的数据库特性依赖，支持 x86 和 ARM 部署。

### 6.4 运维自治化演进

- **场景**：运维团队希望从手动运维逐步演进到自动化运维，最终实现自治运维。
- **需求**：分阶段实施、策略渐进式开放、人工审批逐步减少。
- **AUTOPS 优势**：AI 能力分 5 阶段 Agentic 渐进开放（分析→只读工具→计划生成→低风险自动执行→高风险人工确认），策略中心支持灰度发布和模拟验证。

### 6.5 多团队运维协同

- **场景**：运维、安全、开发、管理层需要统一的运维平台协同工作。
- **需求**：统一视图、权限隔离、工单协同、审计追溯。
- **AUTOPS 优势**：RBAC/ABAC 权限模型、内置工单系统、全链路关联（告警↔工单↔执行↔日志↔AI 分析）、统一审计日志。

### 6.6 第一批标准化验证场景

以下场景必须通过平台通用能力完成闭环验证，不允许为任何场景写旁路逻辑：

1. Linux 磁盘空间异常。
2. Windows 服务异常。
3. Web 服务端口不可达。
4. 数据库连接异常或连接数过高。
5. SSL 证书即将过期。
6. 采集器离线。
7. 自动化执行失败。

## 7. 非目标（明确什么不是 AUTOPS 的目标）

### 7.1 不是公有云 SaaS 平台

AUTOPS 不提供 SaaS 服务，不依赖公有云基础设施，不收集用户数据。核心设计目标是私有化部署和离线运行。

### 7.2 不是通用 CI/CD 平台

AUTOPS 不替代 Jenkins、GitLab CI、ArgoCD 等 CI/CD 工具。自动化执行聚焦在运维场景（故障处置、巡检、配置修复、日志清理等），而非应用构建和发布流水线。

### 7.3 不是通用日志分析平台

AUTOPS 不是 ELK/Loki/Splunk 的替代品。日志能力聚焦在运维上下文关联（执行日志、告警关联、AI 诊断证据），而非通用日志采集、分析和可视化。

### 7.4 不是通用项目管理平台

AUTOPS 不替代 Jira/Tapd 等项目管理工具。工单系统聚焦在运维事件协同（告警转工单、执行失败升级、复盘转知识），而非通用项目管理和任务跟踪。

### 7.5 不是自由 AI Agent

AI 在 AUTOPS 中是受控诊断与决策辅助层，不是自由执行的 Agent。AI 不可绕过权限、审批、策略、安全边界和执行引擎。AI 可以分析、解释、推荐、生成计划和知识草稿，但不能直接执行高风险动作。

### 7.6 不是监控数据长期存储方案

AUTOPS 使用 VictoriaMetrics/Prometheus 存储时序指标，但不追求替代专业时序数据库作为长期数据仓库。指标保留策略由用户通过时序库配置决定。

### 7.7 不追求功能大而全

AUTOPS 不尝试覆盖所有 IT 管理领域。聚焦在运维自治闭环，不涉及 IT 资产采购管理、IT 预算管理、IT 服务目录（ITSM 全集）等超出运维范畴的领域。

### 7.8 不围绕 MVP 做专项开发

MVP 不是项目目标，也不是开发中心。MVP 只用于验证架构闭环是否真实可运行。所有验证场景必须通过平台通用能力完成。

## 8. 成功指标

### 8.1 架构完整性指标

| 指标 | 目标 |
|------|------|
| 九类统一模型覆盖率 | 所有功能归入统一资源/配置/状态/事件/策略/执行/日志/知识/治理模型 |
| 领域边界清晰度 | 每个领域有明确职责、输入输出边界、依赖关系文档 |
| API 规范性 | 所有 API 遵循统一响应结构、错误码、trace_id |
| 代码可追溯性 | 每个重要模块、接口、表、事件、策略、执行动作可在文档中找到设计依据 |

### 8.2 闭环验证指标

| 指标 | 目标 |
|------|------|
| 核心闭环可运行 | 资源发现→资产纳管→配置治理→状态采集→事件识别→告警收敛→AI 诊断→策略决策→自动化执行→实时日志→结果验证→回滚/升级→知识沉淀 全链路可运行 |
| 标准验证场景通过数 | 第一批 7 个标准场景全部通过闭环验证 |
| 告警到处置平均时间 | 相比人工处置缩短 60%+ |
| 告警收敛率 | 告警数量相比原始事件减少 70%+ |

### 8.3 安全与可靠性指标

| 指标 | 目标 |
|------|------|
| 凭证明文暴露 | 零（所有凭证加密存储，导出不含明文） |
| AI 绕过安全边界 | 零（AI 不可绕过策略中心、审批机制和执行引擎） |
| 执行审计覆盖率 | 100%（所有自动化执行有 execution_id、审计日志、实时日志） |
| 高危动作审批率 | 100%（生产环境高危动作必须人工确认） |
| 执行失败回滚率 | 100%（失败执行有回滚动作或升级工单） |

### 8.4 部署与兼容性指标

| 指标 | 目标 |
|------|------|
| 私有化部署 | 支持单机、Docker Compose、多节点、物理机四种部署模式 |
| 离线部署 | 完整支持，不依赖外部网络 |
| 数据库兼容 | MySQL/MariaDB 默认，保留 PostgreSQL、openGauss、人大金仓、达梦迁移空间 |
| 硬件架构 | 支持 x86 和 ARM |
| 模型降级 | 大模型不可用时，告警、采集、策略和自动化不中断，AI 分析显示不可用 |

### 8.5 知识与演进指标

| 指标 | 目标 |
|------|------|
| 知识库覆盖率 | 标准处置方案覆盖第一批 7 类验证场景 |
| 策略优化闭环 | 处置结果可反馈至策略中心优化策略 |
| 工单复盘转知识 | 关闭工单可生成知识草稿并进入审核流程 |
| 文档与代码一致性 | 代码变更同步更新文档，文档过期归档 |

### 8.6 验收标准（引用自全局约束）

平台是否完成，不看代码量，不看页面数量，而看是否满足：

1. 架构是否一致。
2. 领域边界是否清晰。
3. 数据模型是否可演进。
4. API 是否规范。
5. 前端流程是否符合运维真实工作。
6. 是否能自动发现问题。
7. 是否能自动分析问题。
8. 是否能安全执行处置动作。
9. 是否有实时日志。
10. 是否能验证结果。
11. 是否能失败回滚或升级工单。
12. 是否能沉淀知识。
13. 是否有审计和安全边界。
14. 是否能私有化部署、离线部署和升级回滚。
15. 是否能通过自动化测试验证核心链路。

## 附录：术语定义

| 术语 | 定义 |
|------|------|
| 自治运维 | 系统能够自动完成从感知到处置的闭环，减少人工介入，但高风险动作仍需人工确认 |
| 私有化部署 | 平台完全部署在用户自有的基础设施上，不依赖公有云服务 |
| 离线部署 | 在无外网连接的环境中完成安装、运行和升级 |
| 国产化兼容 | 支持国产数据库、国产操作系统、国产硬件平台 |
| 统一模型 | 平台底层由九类模型（资源、配置、状态、事件、策略、执行、日志、知识、治理）驱动，任何功能必须归入其中 |
| 策略驱动 | 自动化执行动作由策略中心统一调度和约束，而非脚本直接触发 |
| 闭环验证 | 从发现到知识沉淀的完整链路必须真实可运行、可验证、可审计 |
