# AUTOPS 信息架构与概念重构方案（2026-06-27）

目标：消除“概念不清、职责重叠、命名碰撞、放错模块”，让每个模块定位聚焦、每个页面兑现定位。
本方案先把**概念模型**定清楚，再给**目标信息架构**，最后给**分步实施计划**。
实现统一沿用既有模式：被并页加 `embedded` prop、宿主 `el-tabs(lazy)`、旧路径 `redirect ?tab=`。

---

## 一、核心概念模型（语义层，先定义后落地）

当前最大的语义债是 **事件 / 告警 / 异常 / 故障** 四个概念边界模糊、且 `异常↔告警` 双向纠缠
（代码现状：state→critical 建 Anomaly；巡检发现建 Anomaly；而 Anomaly 又反向“通知告警中心”）。

### 定为单向四层（推荐，理由见下）

| 概念 | 英文 | 定义 | 面向 | 归属 |
|---|---|---|---|---|
| 事件 | Event | 原始遥测/采集/状态变更，只读事实流 | “发生了什么（原始）” | M4 事件流 |
| 告警 | Alert | 对事件/指标按**规则/阈值**判定出的信号，支持去重/收敛/静默/确认 | “哪些信号值得关注” | M4 告警 |
| 异常 | Anomaly | 当一个/多个告警或检测**需要人介入**时收敛成的**待处置案例**（带 assignee、生命周期、关联工单） | “要处理哪件事” | M4 异常 → M5 处置 |
| 故障处置 | Incident | 针对异常的**处置闭环**（诊断→影响→分级→建议→验证→关闭） | “怎么处理” | M5 故障工作台 |

**数据流（严格单向）**：`Event → Alert →(收敛)→ Anomaly → Incident(处置) → Ticket(协同) → Knowledge(沉淀)`

### #5 最佳建议：保留“告警 vs 异常”分层，但改为严格单向

- **不合并**。分层的价值是真实的：告警层做**去重/收敛/静默**（N 条告警），异常层做**案例化处置**（收敛成 1 件事 + 指派 + 生命周期）。这与业界 Alert vs Incident/Problem 的分层一致，合并会丢掉“多告警→一案例”的收敛能力。
- **但必须消除双向纠缠**：删除 `Anomaly→通知告警中心` 的反向链路；统一为 `Alert→(规则/收敛)→Anomaly`。
- **落地**：M4 内用分组明确“①事件流 ②告警(信号) ③异常(案例)”三段递进；异常的“历史/记录”由系统自动沉淀，不另立菜单。
- **收益**：用户一眼看懂“原始→信号→案例→处置”的递进，不再问“告警和异常是不是一回事”。

## 七、分步实施计划（按风险从低到高）

> 全前端步骤沿用 embedded/tab/redirect 模式，旧路径不失效；后端步骤单独隔离。

- **步骤 1（低险·纯前端）重复合并**：处置模板→剧本库 tab；安全基线→redirect 基线巡检。
- **步骤 2（低险·纯前端）AI 诊断归位**：M5 用真 AiDiagnosisPage，退役薄面板；M8 移除“AI诊断记录”。
- **步骤 3（低险·前端）M8 净化 + AI 设置组**：Prompt模板/AI工具策略/模型服务 归「AI 设置」(挂 M11)；规则缺口→M6“规则覆盖度”；M8 只剩知识。
- **步骤 4（低险·前端）命名消歧**：Agent管理→采集节点；配置事实→配置快照；M3 话术统一。
- **步骤 5（中险·前端）M9 收敛**：成品报告按类型 tab；三子区重排。
- **步骤 6（中险·前端）M12 分组**：3 子组视觉分组。
- **步骤 7（高险·后端，需评审）告警/异常单向化**：删 Anomaly→Alert 反向通知，统一 Event→Alert→Anomaly→Incident；前端 M4 三段分组。

每步独立提交、可回滚；步骤 1–6 可连续做，步骤 7 单独排期并做事件链回归。

**#5 落地结论**：采用「分层 + 严格单向」。`anomaly/handlers.py` 重构为
`检测源(StateCritical/巡检) + 收敛源(高/严重 Alert) → ANOMALY_DETECTED → 落库 Anomaly`，
移除旧的 `on_anomaly_detected→ALERT_CREATED` 反向链路。告警引擎 (`Event→规则→Alert`) 不变。
